التفاصيل التقنية والمعمارية لمشروع Mazzady

قمت بتصميم وتطوير هذه المنصة بجهد فردي كامل لتكون بيئة مزادات إلكترونية شاملة، حيث تطلبت دقة متناهية في الأداء والأمان ومعالجة البيانات اللحظية. وفيما يلي تفصيل شامل للبنية التقنية، الخوارزميات، والتحديات التي قمت بحلها.

أولاً: البنية التقنية (Architecture & Stack)
1. واجهة المستخدم (Frontend): اعتمدت على إطار عمل Angular 19، مستخدماً معمارية Standalone Components لتقليل الاعتماديات وتسريع التحميل. كما وظفت تقنية Signals لإدارة الحالة الحركية (State Management) بكفاءة عالية وبدون استهلاك مفرط للذاكرة، مع دمج حركات تفاعلية (Animations) معقدة لتحسين تجربة المستخدم.
2. الخادم (Backend): قمت ببناء السيرفر باستخدام NestJS 10 لضمان معمارية نظيفة وقابلة للتوسع (Modular Architecture) مبنية بالكامل على TypeScript.
3. قاعدة البيانات (Database): استخدمت MongoDB مع Mongoose للتعامل مع كميات ضخمة من البيانات غير المهيكلة بسرعة فائقة، خاصة بيانات المزايدات المتلاحقة.
4. الاتصال اللحظي (Real-time): وظفت WebSockets لإدارة شريط التزايد اللحظي والمزادات المباشرة، مما يضمن تحديث الأسعار لدى جميع المستخدمين في أجزاء من الثانية دون الحاجة لتحديث الصفحة.
5. التخزين السحابي (Cloud Storage): قمت بدمج خدمة Cloudinary لمعالجة وتخزين صور المنتجات والمستندات الرسمية (كبطاقات الهوية).

ثانياً: التفاصيل الفنية الدقيقة للأنظمة الفرعية
1. خوارزمية التسجيل والتحقق (Authentication & Onboarding):
   - صممت واجهة تسجيل معقدة مقسمة إلى 8 مراحل تفاعلية.
   - التحقق الصارم من الرقم القومي المصري: برمجة خوارزمية للتحقق من أن الرقم يتكون من 14 خانة، ويبدأ بـ 2 أو 3 (حسب سنة الميلاد)، مع ضمان عدم تكراره في قاعدة البيانات.
   - معالجة الصور: فرض قيود على حجم الصور (بحد أقصى 2 ميجابايت لصورة الوجه والخلفية) لتخفيف العبء على الشبكة.
   - التحقق عبر البريد: نظام إرسال رموز تحقق (OTP) عبر البريد الإلكتروني (Nodemailer) بصلاحية زمنية صارمة تبلغ 10 دقائق.
2. محرك المزايدة التلقائية (Auto-Bid Engine): قمت بتطوير خوارزمية تسمح للمستخدم بتحديد حد أقصى للمزايدة، ليقوم الخادم بالمزايدة نيابة عنه بزيادات تدريجية عند وجود منافسين، مع ضمان الحماية من تعارض البيانات (Race Conditions) على مستوى قاعدة البيانات.
3. التنوع في أنظمة المزادات: تصميم هياكل بيانات تدعم أنواعاً مختلفة من المزادات، مثل: المزادات السريعة (Flash Auctions)، المزادات العكسية (Reverse Auctions)، والمزادات الخاصة بالمجموعات (Private Auctions) المحمية برموز مرور.
4. نظام المحافظ ونقاط الولاء (Loyalty Points): نظام لتتبع تفاعلات المستخدم ومشترياته وتحويلها لنقاط يمكن إعادة استخدامها.
5. حماية المسارات (Interceptors & Guards): إضافة معترضات (Interceptors) في الواجهة الأمامية لإرفاق رموز المصادقة (JWT) تلقائياً مع كل طلب، وبناء حراس (Guards) في الخادم للحماية المتقدمة.

ثالثاً: التحديات التقنية والحلول المعمارية
1. تجاوز قيود نظام الملفات (File System) في بيئة Vercel (Serverless):
   - المشكلة: بيئة Vercel لا تدعم حفظ الملفات محلياً في مجلد `uploads/`.
   - الحل: قمت بإعادة هيكلة نظام معالجة الملفات في NestJS ليقوم بالتقاط الصور وتخزينها مؤقتاً في الذاكرة الحية (MemoryStorage)، ثم استخدام مجاري البيانات (Streams) لرفعها مباشرة إلى Cloudinary دون ملامسة القرص الصلب، وحفظ الرابط الآمن فقط في قاعدة البيانات.
2. معالجة التوجيه الصلب (Hardcoded URLs):
   - المشكلة: وجود مسارات ثابتة (مثل localhost) في أكثر من 20 ملفاً بالواجهة الأمامية.
   - الحل: قمت بتغليف جميع الروابط وتحويلها لمتغيرات بيئة ديناميكية عبر ملفات `environment.ts` للتكيف التلقائي مع بيئات التطوير والإنتاج.
3. منع تعارض البيانات (Race Conditions) في التحقق غير المتزامن (Async Validation):
   - المشكلة: كتابة المستخدم السريعة لحقول فريدة (مثل اسم المستخدم) كانت تغرق الخادم بطلبات متتالية، مما يسبب أخطاء غير متوقعة.
   - الحل: طبقت تقنيات `DebounceTime` و `DistinctUntilChanged` من مكتبة RxJS في Angular لتأخير إرسال الطلب حتى يتوقف المستخدم عن الكتابة لمدة 500 ملي ثانية.
4. تكامل نظام الدخول الموحد (OAuth):
   - المشكلة: مستخدمو Google و Facebook لا يمتلكون حقول كلمة المرور ورقم الهاتف في الحزمة المستلمة، مما أدى لرفض قاعدة البيانات للعملية.
   - الحل: قمت بتعديل المخطط (Schema) لتصبح هذه الحقول اختيارية، مع إضافة مؤشرات (Flags) تحدد آلية تسجيل دخول المستخدم.
5. الأمان ومكافحة الهجمات الدخيلة (Security & Anti-DDoS):
   - الحل: قمت بضبط إعدادات CORS وتطبيق مقيد الطلبات (`@nestjs/throttler`) لحماية نقاط النهاية الحساسة من هجمات حجب الخدمة ومحاولات التخمين الآلي.

رابعاً: نموذج لتعقيدات الكود (حل قيود Serverless)
لتوضيح المعالجة المتقدمة للبيانات، هذا هو الجزء المسؤول عن استخدام Streams لنقل بيانات الصور من الذاكرة إلى منصة التخزين السحابية دون تخزين محلي:

```typescript
import { v2 as cloudinary } from 'cloudinary';
import { Injectable } from '@nestjs/common';

cloudinary.config({
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
  api_key: process.env.CLOUDINARY_API_KEY,
  api_secret: process.env.CLOUDINARY_API_SECRET,
});

@Injectable()
export class ImageCompressionService {
  async uploadImage(fileBuffer: Buffer, folder: string): Promise<string> {
    return new Promise((resolve, reject) => {
      // فتح مجرى رفع بيانات مباشر مع Cloudinary
      const uploadStream = cloudinary.uploader.upload_stream(
        { folder: `mazzady/${folder}` },
        (error, result) => {
          if (error) return reject(error);
          resolve(result.secure_url);
        }
      );
      
      // تحويل حاوية الذاكرة (Buffer) إلى مجرى قراءة (Stream) لمنع التخزين المؤقت على الخادم
      const stream = require('stream');
      const bufferStream = new stream.PassThrough();
      bufferStream.end(fileBuffer);
      bufferStream.pipe(uploadStream);
    });
  }
}
```
